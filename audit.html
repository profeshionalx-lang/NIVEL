1. Резюме

Проект — MVP коучинговой платформы для падел-тенниса с чёткой ролевой моделью (ученик / тренер), адекватной схемой Postgres и разумным использованием Server Components + Server Actions. Архитектурно всё ровно: auth через @supabase/ssr, RLS включён на всех пользовательских таблицах, маршрутизация защищена через proxy.ts.

Однако есть один критический баг-блокер (создание цели падает из-за рассинхрона схемы и кода), несколько высоких (гонка session_number, N+1 запросы, RLS-дыра на удаление), и много среднего/мелкого — типизация через Record<string, unknown>, дублирование auth-проверок, мёртвые ссылки в навигации, отсутствие тестов, отсутствие next/image.

2. Критические проблемы

highСхема в репозитории ≠ схема в проде (но код с проде совпадает)
supabase/migrations/*.sql vs live Supabase project

Проверено вживую 2026-04-17 через Supabase REST. Прод-база уже приведена в соответствие с кодом:

  • goals.custom_problem — есть ✅ (в миграциях 001/002 в репо — нет)
  • goals.session_count — удалена ✅ (в миграции 001 — есть и NOT NULL)
  • RPC search_problems — существует ✅ (в миграциях вообще нет)

Это значит, что в Supabase выполняли DDL руками (через SQL editor) помимо коммитов в supabase/migrations/. Приложение в проде работает. Но: кто-то, кто развернёт новый Supabase-проект с нуля по файлам из репо, получит сломанное приложение.

Что делать. Добавить недостающую миграцию 003_goals_and_search.sql в репо — чтобы npx supabase db reset и новые окружения собирались корректно:

ALTER TABLE public.goals
  DROP COLUMN IF EXISTS session_count,
  ADD COLUMN IF NOT EXISTS custom_problem TEXT;

CREATE OR REPLACE FUNCTION public.search_problems(query_text TEXT)
RETURNS TABLE(id INT, name TEXT, category_name TEXT)
LANGUAGE sql STABLE AS $$
  SELECT p.id, p.name, c.name AS category_name
  FROM public.problems p
  JOIN public.problem_categories c ON c.id = p.category_id
  WHERE p.name ILIKE '%' || query_text || '%'
  ORDER BY p.sort_order
  LIMIT 20;
$$;
critГонка при создании занятия: неуникальный session_number
src/lib/actions/sessions.ts:39–48 · таблица sessions

createSession() считает следующий номер через count + 1 и пишет инсертом — это классический race condition. Два параллельных вызова (два устройства тренера, двойной клик, прокси-ретрай) получат одинаковый session_number, и оба инсёрта пройдут, потому что на (goal_id, session_number) нет UNIQUE.

Что делать. Либо держать счётчик в БД и считать атомарно, либо добавить уникальность и ретрай:

ALTER TABLE public.sessions
  ADD CONSTRAINT sessions_goal_number_uniq UNIQUE (goal_id, session_number);

Лучший вариант — вынести всю логику createSession в Postgres-функцию (SECURITY DEFINER), чтобы всё произошло в одной транзакции и без сетевых roundtrip’ов (их сейчас до 10+).

3. Важные проблемы

highcreateSession — десяток отдельных запросов без транзакции
src/lib/actions/sessions.ts

На одно занятие из 3 упражнений по 2 скилла — до ~15 раундтрипов в Supabase, каждый с overhead’ом REST-слоя. При падении в середине (например, на session_exercise_skills) остаются «полу-занятия»: строка в sessions создана, скиллы — частично, прогресс — частично. Supabase JS не даёт многооператорных транзакций из клиента.

Что делать. Один SQL-функционал create_session(goal_id, exercises jsonb), SECURITY DEFINER, внутри — CTE и ON CONFLICT. Вызов из server action — единственный rpc().

highN+1 запросы на дашборде и списке учеников
src/app/dashboard/page.tsx:35–62 · src/app/trainer/students/page.tsx:39–61 · src/app/trainer/students/[id]/page.tsx:48–75

На каждую цель — отдельный count-запрос к sessions. На каждого ученика — два count’а. При 20 учениках по 3 цели — ~80 сетевых вызовов на одну страницу.

Что делать. Сделать view или RPC, которая одним запросом отдаёт агрегаты:

CREATE VIEW public.goals_with_counts AS
SELECT g.*, COALESCE(s.cnt, 0) AS completed_sessions
FROM public.goals g
LEFT JOIN (
  SELECT goal_id, COUNT(*) AS cnt FROM public.sessions GROUP BY goal_id
) s ON s.goal_id = g.id;
highRLS не покрывает DELETE и UPDATE для ряда таблиц
supabase/migrations/001_initial_schema.sql

RLS включён, но политики заданы только для SELECT/INSERT (и частично UPDATE). По умолчанию в Postgres при включённом RLS и отсутствии политики операция запрещена всем — это, кстати, спасает от явных утечек, но одновременно означает, что ни тренер, ни ученик физически не могут удалить цель, занятие, упражнение или скилл, даже если продукт захочет такую фичу.

Более тонкий момент: политика "Goal problems insert" требует goals.user_id = auth.uid(), значит тренер не может добавить проблему к цели ученика. Сейчас это и не делается, но стоит явно зафиксировать намерение.

Что делать. Явно добавить политики DELETE там, где они нужны, и UPDATE для goals/sessions с ролью trainer:

CREATE POLICY "Goals delete own" ON public.goals
  FOR DELETE USING (user_id = auth.uid() OR public.is_trainer());
highproxy.ts: любой throw → принудительный logout
src/proxy.ts:55–60

Внешний try/catch при любой ошибке (сетевая, 5xx от Supabase, таймаут) редиректит на /login. Пользователь видит логаут при кратком сбое Supabase. Также это маскирует баги — прод будет выбрасывать людей, а вы не узнаете.

Что делать. При сетевых ошибках возвращать NextResponse.next() и логировать; редирект на login делать только при явном отсутствии сессии.

highproxy.ts + каждая страница — дублированный getUser()
src/proxy.ts · src/app/dashboard/layout.tsx · src/app/dashboard/page.tsx · далее везде

На один запрос дашборда выполняется ≥3 вызова supabase.auth.getUser() (proxy → layout → page), каждый идёт в Supabase Auth API. Плюс каждая страница заново читает profiles. Суммарно ~200–400 мс лишнего latency.

Что делать. Прочитать профиль один раз в dashboard/layout.tsx и прокинуть через React.cache() или контекст серверных компонентов. Либо завести lib/auth.ts с cache(getUserWithProfile).

4. Средние проблемы

medТипизация фактически отключена: Record<string, unknown> по всему коду
src/app/dashboard/page.tsx · src/app/trainer/students/page.tsx · /sessions/[id] · library/page.tsx

На всех страницах результаты Supabase приводятся к Record<string, unknown> и далее «as string» / «as number». Это визуальный TypeScript без реальной проверки: опечатка в имени колонки проявится только в рантайме.

Что делать. Сгенерировать типы:

npx supabase gen types typescript --project-id <id> > src/lib/supabase/database.types.ts

И параметризовать клиент: createServerClient<Database>(...). Всё остальное начнёт выводиться само.

medДублированная auth-boilerplate в каждой защищённой странице
dashboard/layout.tsx, dashboard/page.tsx, trainer/*, sessions/[id]

Паттерн getUser → redirect → profiles.select → role check повторяется 6 раз почти слово в слово. Любая логика (например, добавить trial-период) расползётся на 6 правок.

Что делать. Одна утилита:

// src/lib/auth.ts
export const requireTrainer = cache(async () => {
  const supabase = await createClient();
  const { data: { user } } = await supabase.auth.getUser();
  if (!user) redirect("/login");
  const { data: profile } = await supabase
    .from("profiles").select("*").eq("id", user.id).single();
  if (!profile || profile.role !== "trainer") redirect("/dashboard");
  return { user, profile };
});

6. Безопасность и RLS

Хорошо:

  • RLS включён на всех пользовательских таблицах, политики логичные.
  • Тренерская проверка сделана двухуровнево: и в server action (profile.role === "trainer"), и в RLS (is_trainer()).
  • PKCE-flow Supabase OAuth, получение сессии через exchangeCodeForSession.
  • handle_new_user и increment_skill_progress — SECURITY DEFINER, что корректно ограничивает привилегии.

Плохо / требует внимания:

  • Нет DELETE-политик (см. раздел High). При желании добавить «удалить цель/занятие» — не получится.
  • Тренер видит email любого ученика (политика profiles SELECT). Это ожидаемо, но стоит зафиксировать в доках — по GDPR нужна правовая основа.
  • handle_new_user не обрабатывает email-коллизии и не поддерживает обновление avatar_url при повторном логине. Если Google сменил аватар — он не подтянется в profiles.
  • Нет rate-limiting на server actions. createSession вставляет в 5 таблиц — злоумышленник с валидной trainer-сессией может за минуту создать миллион пустых строк.
  • Нет CSP / security headers. Стоит добавить Strict-Transport-Security, Content-Security-Policy, Referrer-Policy через headers() в next.config.ts.
  • В handle_new_user падение триггера (например, если profiles.email NOT NULL нарушен на null-email) откатит весь signup. Низкая вероятность, но стоит обернуть в BEGIN/EXCEPTION.

8. Типизация и код-стайл

  • Главное: сгенерированные типы Supabase (Database) — даст типобезопасность на всех запросах без каких-либо as.
  • Удалить из репозитория Record<string, unknown> — по факту это any в костюме.
  • Button.tsx и Card.tsx объявлены, но не используются. Либо применить везде, либо удалить.
  • В dashboard/layout.tsx и page.tsx один и тот же TopBar/BottomNav layout-стиль распадается на дочерних страницах (/goals/new, /sessions/[id] рисуют свой header). Это сознательное решение — но стоит явно выделить «full-screen» и «tabbed» layout через route groups app/(tabs)/... / app/(fullscreen)/....
  • В handleLogout — router.push("/login") + router.refresh(). При включённом RSC-кеше достаточно window.location.href = "/login" или router.replace("/login").
  • В LoginPage нет состояния ошибок при failed OAuth (query ?error=auth из callback не читается).

9. UX и навигация

  • Сломанные табы — см. выше: /sessions не существует; «Цели» ведёт в форму; подсветка активного не работает.
  • Нет страницы «Мои цели» — только карточки на дашборде. Нет способа отменить цель из UI, хотя action cancelGoal написан.
  • Нет возврата к сессии после написания инсайта — router.refresh() перерисовывает страницу, но фокус теряется.
  • Нет indicator «тренер уже написал заметки» в списке занятий у ученика.
  • Мобильный focus-ring на textarea/input подразумевается, но кнопка «Создать цель» подсвечивается active:scale без ARIA-атрибутов; pressed-state не заметен для assistive tech.
  • Keyboard UX: в трейнерской форме Enter в поле скилла добавляет его — хорошо. Но в поле «название упражнения» Enter ничего не делает. Логично — добавлять упражнение.
  • Пустой state для тренера отсутствует: если учеников нет — он видит пустой список (правильно), но если нет целей у выбранного ученика — нельзя создать цель за него (должна быть такая фича, и в текущей RLS её нет — см. High).

10. Инфра и процессы

  • Тестов нет вообще. Минимум — Playwright-тесты критических сценариев: логин-тренера, создание цели учеником, создание занятия тренером, добавление инсайта.
  • CI не настроен (нет .github/workflows). Добавить lint + typecheck + build на PR.
  • Нет vercel.ts. Рекомендован как замена vercel.json, позволяет типизированно задать crons/headers/rewrites.
  • В последних коммитах видно историю с middleware.ts → proxy.ts. Сейчас src/proxy.ts — корректная форма для Next.js 16. Остатки try/catch из debug-коммита 31a5322 всё ещё в коде — см. High про logout-лавину.
  • Нет структурированного логирования (ни в server actions, ни в proxy). console.error("[proxy] error:", ...) — нормальный минимум, но на Vercel хорошо бы отправлять в @vercel/otel.
  • Нет бэкап-политики Supabase, описанной в репо (это не код, но упоминание в README стоит).
  • Vercel CLI у вас 51.4.0, актуальная 51.5.1 — стоит обновить (npm i -g vercel@latest).
